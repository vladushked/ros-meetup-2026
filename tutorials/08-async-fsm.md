# Tutorial 08: Асинхронная FSM

## Проблема
При усложнении автомата переходы начинают происходить в разных местах кода и теряется прозрачность.

## Цель
Собрать состояние и переходы в один объект FSM и исполнять их централизованно в async loop.

## Файлы примера
- `ros-meetup-2026/examples/async_fsm/task_fsm.py`
- `ros-meetup-2026/examples/async_fsm/fsm_driver_node.py`

## Что важно
1. FSM на `transitions.AsyncMachine`.
2. Внешний код ставит только `pending transition`.
3. `step()` проверяет валидность и выполняет переход.
4. Нода публикует текущее состояние только после фактического изменения.

## Мини-сценарий из примера
`CHECKS -> READY -> EXECUTION -> STOP`.

## Почему это решение принято
FSM перестает быть «распыленной логикой» и превращается в проверяемый контракт состояний/переходов.
