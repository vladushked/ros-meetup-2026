# Tutorial 07: Паттерн `register_loop`

Соответствует второй половине слайда 20 из [presentation.md](../presentation.md).

## Проблема

Когда в одной ноде одновременно живут `grasping`, `fsm`, `watchdog`, heartbeat и ещё несколько фоновых задач, ручной orchestration быстро становится шумным.

## Цель

Автоматически регистрировать loop-методы и запускать их единым раннером.

## Файл примера

- [examples/register_loop/task_orchestrator_node.py](../examples/register_loop/task_orchestrator_node.py)

## Что показывает пример

1. Декоратор `register_loop` помечает loop-методы.
2. Класс собирает такие методы в `async_loops`.
3. `run()` запускает их параллельно через `asyncio.wait(...)`.

В примере выделены отдельные циклы:
- `ros_loop`
- `grasping_loop`
- `fsm_loop`
- `watchdog_loop`

## Почему этот паттерн удобен

- каждая фоновая задача остаётся локальной и читаемой;
- добавление нового воркера сводится к одному `@register_loop`;
- orchestration не приходится переписывать при росте ноды.

## Почему это решение принято

Так проще масштабировать сложную сервисную ноду без возврата к большому callback-mesh.
